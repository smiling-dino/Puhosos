# ROSBridge для ветки `bugfix_v4`

## Что исправлено

В `RobotBrain.cs` базового коммита `884f7e0` уже есть экспериментальный блок ROS, но он решает обратную задачу:

- принимает `/gfsx/cmd_vel` из ROS и управляет симулятором;
- публикует виртуальную телеметрию Unity в ROS.

Для запуска обученной модели на физическом GFS-X нужен другой поток:

`ML-Agents policy -> Unity actuators -> ROS topics -> Raspberry Pi -> motors/servos`.

`Assets/Scripts/ROSBridge.cs` реализует именно этот поток и не требует менять `RobotBrain.cs`.

## Почему мост читает контроллеры, а не приватные actions

`ROSBridge` берет уже примененные команды из:

- `TrackController` — после dead zone и ограничения изменения PWM;
- `GripperController` — с актуальным открытием/закрытием;
- `GripperController.LiftNormalized` — с текущим складыванием руки буквой Z;
- поворота `cameraServo`.

Так физический робот получает то же управляющее состояние, которое реально отрабатывает цифровой двойник.

## Установка

1. Переключитесь на ветку с настройкой:

```powershell
git switch bugfix_v4
```

2. Откройте проект в Unity и дождитесь компиляции.

3. Выберите **одного** робота, на котором будет запускаться модель в режиме инференса.

4. На корневом объекте робота уже должен быть компонент `ROSBridge`. Если вы переносите настройку в другую сцену, добавьте его вручную.

5. В `RobotBrain` оставьте старый флаг **Enable Ros Bridge = false**. Старый встроенный мост предназначен для управления симулятором из ROS и конфликтует по топикам с новым выходным мостом.

6. В `Behavior Parameters`:

- назначьте обученную ONNX-модель;
- установите `Behavior Type = Inference Only`.

7. В `ROSBridge`:

- назначьте `Robot Brain`, `Track Controller`, `Gripper Controller`;
- назначьте `Camera Servo`, если автоматический поиск выбрал не тот объект;
- выставьте реальные пределы скорости робота;
- включите `Bridge Enabled`.

8. В Unity откройте `Robotics -> ROS Settings` и задайте IP Raspberry Pi/ROS-машины и порт ROS-TCP Endpoint.

## Контракт топиков

| Топик | Тип | Значение |
|---|---|---|
| `/gfsx/cmd_vel` | `geometry_msgs/Twist` | `linear.x` в м/с, `angular.z` в рад/с |
| `/gfsx/camera_cmd` | `std_msgs/Float32` | нормализованная скорость камеры `[-1; 1]` |
| `/gfsx/gripper_cmd` | `std_msgs/Int32` | `1` — закрыть, `2` — открыть |
| `/gfsx/lift_cmd` | `std_msgs/Int32` | `0` — стоп, `1` — сложить Z, `2` — разложить |

## Защита

В мосте есть:

- EMA-сглаживание движения;
- мгновенный hard stop при нулевой команде;
- запрет публикации при подключенном `mlagents-learn`;
- блокировка второго издателя при клонировании арен;
- watchdog: если `RobotBrain.StepCount` не обновляется более `0.5` секунды, публикуется остановка;
- остановка при паузе, выключении компонента и завершении приложения.

Unity-watchdog не защищает от полного зависания процесса или обрыва сети, потому что в этих случаях Unity уже не может отправить нулевую команду. На Raspberry Pi/ROS-ноде обязателен второй watchdog: если новый `/gfsx/cmd_vel` не получен за `0.5` секунды, моторы должны быть остановлены локально.

## Проверка перед реальным запуском

1. Поднимите ROS-TCP Endpoint.
2. Запустите `ros2 topic echo /gfsx/cmd_vel`.
3. Включите Unity с поднятыми над полом гусеницами.
4. Убедитесь, что:
   - при движении модели меняются `linear.x` и `angular.z`;
   - при остановке оба значения становятся строго `0`;
   - после паузы Unity приходит нулевая команда;
   - закрытие/открытие клешни дает `1/2`;
   - складывание руки дает `1`, остановка — `0`, раскладывание — `2`.
5. Только после этого ограничьте скорость на ROS-стороне и поставьте робота на пол.
